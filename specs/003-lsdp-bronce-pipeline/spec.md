# Especificacion de Feature: Incremento 3 - Lakeflow Spark Declarative Pipelines - Medalla de Bronce

**Branch de Feature**: `003-lsdp-bronce-pipeline`
**Creado**: 2026-04-04
**Estado**: Draft
**Entrada**: Incremento 3 del proyecto DbsFreeLakeflowSparkDeclarativePipelinesBase — Crear el pipeline Lakeflow Spark Declarative Pipelines (LSDP) con el procesamiento exclusivo de la medalla de bronce, usando AutoLoader para ingestión incremental de los parquets AS400, storage dinámico (Volume o Amazon S3) y la tabla Parametros en Unity Catalog como fuente de configuración. Todas las streaming tables deben tener liquid cluster con los campos del cluster ordenados primero mediante la utilidad LsdpReordenarColumnasLiquidCluster. Todo el código debe ser 100% compatible con LSDP y optimizado para Databricks Free Edition Serverless Compute.

## Clarificaciones

### Sesion 2026-04-04

- P: Debe el archivo de definicion del pipeline LSDP (JSON/YAML de configuracion con parametros, catalogo/esquema por defecto y lista de scripts) crearse y versionarse en el repositorio como parte de este incremento? -> R: No. El archivo de definicion del pipeline se configura manualmente desde la UI de Databricks por el ingeniero de datos al momento del despliegue. No es un artefacto del repositorio en este incremento.
- P: En que punto de los scripts LSDP de bronce debe configurarse spark.sql.shuffle.partitions para optimizar el computo Serverless? -> R: No se configura. Los scripts LSDP de bronce no llaman a spark.conf.set("spark.sql.shuffle.partitions"). Se usa el valor por defecto del entorno Serverless de Databricks Free Edition.
- P: Cual es el alcance de las pruebas TDD para el pipeline de bronce? -> R: Las pruebas TDD cubren EXCLUSIVAMENTE las 3 utilidades de `utilities/` (Python puro importable). Los notebooks de `transformations/` NO son testeables via TDD porque ejecutan codigo a nivel de modulo (`spark.conf.get("pipelines.parameters.*")`, `obtener_parametros()`) que requiere un pipeline LSDP desplegado — al importar el modulo, ese codigo se ejecuta y falla fuera del contexto LSDP. La validacion de las transformaciones se realiza unicamente mediante el despliegue del pipeline.
- P: Cual es el modo de inferencia de schema de AutoLoader para los parquets AS400 y como se maneja la evolucion de schema? -> R: Inferencia automatica desde los metadatos del parquet (Opcion A). AutoLoader DEBE tener habilitado schema evolution: si el esquema del parquet cambia (columnas agregadas o eliminadas), el pipeline debe seguir procesando los datos correctamente. Para esto se configura `cloudFiles.schemaEvolutionMode=addNewColumns` y `cloudFiles.schemaLocation` apuntando a una ruta donde AutoLoader almacena el schema inferido y su historial de evoluciones. Cada streaming table recibe como parametro del pipeline la ruta relativa de su schema location; en LSDP estas rutas NO son dbutils.widgets (exclusivos de notebooks), sino parametros del pipeline leidos con `spark.conf.get("pipelines.parameters.X")`. Se agregan tres nuevos parametros del pipeline: `rutaSchemaLocationCmstfl`, `rutaSchemaLocationTrxpfl` y `rutaSchemaLocationBlncfl`, representando las rutas relativas que `LsdpConstructorRutas` convierte en rutas absolutas (Volume o S3) en tiempo de ejecucion.
- P: Cual es el tipo de dato PySpark del campo FechaIngestaDatos y con que funcion se genera? -> R: TimestampType, generado con F.current_timestamp(). Captura la fecha y hora exacta de la ejecucion del pipeline con precision de microsegundos. Este tipo es el correcto para auditorias de ingesta y permite distinguir multiples ejecuciones del mismo dia.
- P: Deben las streaming tables de bronce incluir Data Quality Checks mediante @dp.expect? -> R: No. Las streaming tables de bronce no incluyen ningun decorador @dp.expect, @dp.expect_or_drop ni @dp.expect_or_fail. El dato crudo siempre se persiste tal cual (append-only). Las validaciones de calidad de datos se aplicaran a partir de la medalla de plata en incrementos posteriores.
- P: Cual debe ser la estructura interna de cada script de bronce para permitir el TDD sin infraestructura de AutoLoader? -> R: Cada script expone una funcion pura transformar_<tabla>(df) separada del decorador @dp.table para separacion de responsabilidades. Sin embargo, los notebooks de transformations/ ejecutan codigo a nivel de modulo (patron Closure: `spark.conf.get`, `obtener_parametros()`) que impide su importacion desde TDD. El TDD cubre unicamente las utilidades de `utilities/` (Python puro importable). Las transformaciones se validan mediante despliegue del pipeline LSDP.
- P: Donde debe ejecutarse la lectura de LsdpConexionParametros y la construccion de rutas en el patron Closure con tres scripts separados? -> R: Cada script de bronce es autocontenido: lee sus propios parametros via spark.conf.get(), invoca LsdpConexionParametros y LsdpConstructorRutas de forma independiente a nivel de modulo. No existe modulo inicializador compartido. Cada script puede evolucionar y desplegarse de forma independiente sin acoplamiento entre scripts.
- P: Quien es responsable de validar que las claves requeridas de la tabla Parametros esten presentes: la utilidad LsdpConexionParametros o cada script consumidor? -> R: LsdpConexionParametros solo lee y retorna el diccionario completo sin validar claves (es una utilidad generica reutilizable). Cada script de bronce valida las claves que necesita al momento de usarlas; si una clave requerida falta en el diccionario, el script lanza excepcion con el nombre de la clave faltante. Esto sigue el principio de responsabilidad unica: la utilidad lee, el consumidor valida.

## Escenarios de Usuario y Pruebas

### Historia de Usuario 1 - Ingestión Incremental Declarativa de los Parquets AS400 en Bronce (Prioridad: P1)

El ingeniero de datos despliega el pipeline LSDP con la medalla de bronce activa. Al ejecutar el pipeline por primera vez, AutoLoader detecta los archivos parquet existentes de CMSTFL, TRXPFL y BLNCFL en la ruta configurada (Volume o S3), los ingesta y crea las tres streaming tables en el catalogo `bronce`, esquema `regional`: `cmstfl` (72 columnas), `trxpfl` (62 columnas) y `blncfl` (102 columnas), cada una con el campo `FechaIngestaDatos` agregado automaticamente como marca de tiempo de ingesta. En ejecuciones posteriores, AutoLoader detecta únicamente los archivos nuevos y los ingesta de forma incremental, acumulando el historial sin duplicar registros previamente cargados.

**Por que esta prioridad**: La ingestión de los datos fuente AS400 en la medalla de bronce es el cimiento de toda la arquitectura medallion. Sin las streaming tables de bronce correctamente pobladas, las medallas de plata y oro no pueden existir ni procesarse.

**Prueba Independiente**: Se ejecuta el pipeline LSDP apuntando a los parquets generados en el Incremento 2. Se verifica que las tres streaming tables existan en `bronce.regional` con los conteos de registros correctos, que el campo `FechaIngestaDatos` este presente en cada tabla, y que al agregar nuevos parquets y re-ejecutar el pipeline, solo los registros nuevos sean ingresados (sin duplicados de los anteriores).

**Escenarios de Aceptacion**:

1. **Dado** que los parquets AS400 del Incremento 2 existen en la ruta configurada y las streaming tables de bronce aún no existen, **Cuando** se ejecuta el pipeline LSDP por primera vez, **Entonces** se crean las tres streaming tables (`cmstfl`, `trxpfl`, `blncfl`) en `bronce.regional` con todos los registros de los parquets más el campo `FechaIngestaDatos`.
2. **Dado** que las streaming tables de bronce ya contienen datos de una ejecución previa, **Cuando** se agregan nuevos parquets AS400 a la ruta y se re-ejecuta el pipeline, **Entonces** solo los registros de los nuevos parquets son ingresados; los registros previos se mantienen sin duplicación.
3. **Dado** que el pipeline se ejecuta con `TipoStorage=Volume`, **Cuando** AutoLoader lee los parquets, **Entonces** las rutas de lectura siguen el formato `/Volumes/<catalogo>/<esquema>/<volume>/<ruta_relativa>` construidas dinamicamente desde la tabla Parametros.
4. **Dado** que el pipeline se ejecuta con `TipoStorage=AmazonS3`, **Cuando** AutoLoader lee los parquets, **Entonces** las rutas de lectura siguen el formato `s3://<bucket>/<ruta_relativa>` construidas dinamicamente desde la tabla Parametros.
5. **Dado** que una de las rutas de parquets apuntadas por el pipeline no contiene archivos todavia, **Cuando** se ejecuta el pipeline, **Entonces** AutoLoader espera la presencia de archivos futuros sin lanzar error, manteniendo la streaming table vacía pero funcional.

### Historia de Usuario 2 - Liquid Cluster con Columnas Correctamente Ordenadas (Prioridad: P1)

El ingeniero de datos necesita que cada streaming table de la medalla de bronce tenga su liquid cluster activo y que los campos que componen el cluster sean siempre las primeras columnas del schema de la tabla. Para garantizar esto, la utilidad `LsdpReordenarColumnasLiquidCluster` recibe el DataFrame y la lista de campos del cluster, reordena el DataFrame colocando primero los campos del cluster y luego el resto en su orden original, y retorna el DataFrame reordenado. Esta utilidad es invocada por cada script de transformacion de bronce antes de retornar el DataFrame al decorador LSDP.

**Por que esta prioridad**: El liquid cluster es fundamental para el rendimiento del pipeline y de las consultas posteriores sobre las streaming tables. El ordenamiento correcto de las columnas del cluster es un requisito de calidad de datos definido en el SDD del proyecto.

**Prueba Independiente**: Se invoca la utilidad `LsdpReordenarColumnasLiquidCluster` con un DataFrame de prueba y una lista de campos de cluster. Se verifica que el DataFrame resultante tenga los campos del cluster como las primeras columnas, seguidas del resto en su orden original, y que ninguna columna sea eliminada ni duplicada.

**Escenarios de Aceptacion**:

1. **Dado** que el DataFrame de `cmstfl` tiene 72 columnas y el liquid cluster esta definido para `FechaIngestaDatos` y `CUSTID`, **Cuando** se invoca `LsdpReordenarColumnasLiquidCluster`, **Entonces** el DataFrame resultante tiene `FechaIngestaDatos` como primera columna, `CUSTID` como segunda, y las 70 columnas restantes a continuación en su orden original.
2. **Dado** que el DataFrame de `trxpfl` tiene 62 columnas y el liquid cluster esta definido para `TRXDT`, `CUSTID` y `TRXTYP`, **Cuando** se invoca `LsdpReordenarColumnasLiquidCluster`, **Entonces** el DataFrame resultante tiene `TRXDT`, `CUSTID` y `TRXTYP` como las tres primeras columnas, seguidas de las 59 columnas restantes.
3. **Dado** que el DataFrame de `blncfl` tiene 102 columnas y el liquid cluster esta definido para `FechaIngestaDatos` y `CUSTID`, **Cuando** se invoca `LsdpReordenarColumnasLiquidCluster`, **Entonces** el DataFrame resultante tiene `FechaIngestaDatos` y `CUSTID` como las dos primeras columnas, seguidas de las 100 columnas restantes.
4. **Dado** que se pasa a la utilidad un campo del cluster que no existe en el DataFrame, **Cuando** la utilidad intenta reordenar, **Entonces** se lanza una excepcion explicativa indicando el nombre del campo faltante, deteniendo la ejecucion antes de que el pipeline persista datos con esquema erroneo.

### Historia de Usuario 3 - Utilidades LSDP: Lectura de Parametros y Construccion de Rutas (Prioridad: P1)

El ingeniero de datos verifica que las dos utilidades transversales del LSDP esten disponibles y funcionen correctamente antes del despliegue del pipeline. Ambas utilidades son reutilizables por todos los scripts del proyecto y siguen el patron Closure: sus valores se calculan una sola vez a nivel de módulo al inicializar el pipeline y quedan capturados por closure en las funciones decoradas.

**Por que esta prioridad**: Sin las utilidades de conexión a parametros y construccion de rutas, ningún script del pipeline puede determinar dónde leer los archivos fuente ni cómo conectarse a los datos de configuracion. Son la base de toda la dinamica del proyecto.

**Escenarios de Aceptacion**:

1. **Dado** que la tabla Parametros existe en Unity Catalog con los 11 registros del Incremento 1, **Cuando** se invoca `LsdpConexionParametros` con los datos del catalogo, esquema y nombre de tabla correctos, **Entonces** retorna un diccionario Python con exactamente las claves: `catalogoBronce`, `contenedorBronce`, `TipoStorage`, `catalogoVolume`, `esquemaVolume`, `nombreVolume`, `bucketS3`, `prefijoS3`, `DirectorioBronce`, `catalogoPlata` y `catalogoOro`.

3. **Dado** que el diccionario de parametros tiene `TipoStorage=AmazonS3`, **Cuando** se invoca `LsdpConstructorRutas` con una ruta relativa `LSDP_Base/As400/CMSTFL/`, **Entonces** retorna la ruta `s3://<bucket>/LSDP_Base/As400/CMSTFL/` e imprime la ruta construida en pantalla (el parametro `prefijoS3` NO se utiliza en rutas S3, segun Constitution v1.1.0).

### Casos Borde

- Que sucede cuando la tabla Parametros no existe en Unity Catalog o no tiene los registros requeridos? `LsdpConexionParametros` lanza excepcion si la tabla no existe o no es accesible. Si la tabla existe pero faltan claves especificas, cada script de bronce detecta la clave faltante al intentar usarla y lanza excepcion descriptiva indicando el nombre de la clave que no se encontro en el diccionario.
- Que sucede cuando la ruta de parquets construida por `LsdpConstructorRutas` no es accesible (Volume no existe, bucket S3 sin permisos)? AutoLoader debe reportar el error con un mensaje claro; el pipeline no debe continuar silenciosamente.
- Que sucede cuando `TipoStorage` tiene un valor distinto a `Volume` o `AmazonS3`? `LsdpConstructorRutas` debe lanzar excepcion inmediatamente con los valores validos.
- Que sucede cuando un campo definido como parte del liquid cluster no existe en el DataFrame de un parquet AS400? `LsdpReordenarColumnasLiquidCluster` debe lanzar excepcion explicativa con el nombre del campo faltante.
- Que sucede cuando el pipeline se re-ejecuta múltiples veces con los mismos parquets? AutoLoader debe garantizar que no se dupliquen registros (idempotencia por diseno de AutoLoader).
- Que sucede cuando el entorno Serverless de Databricks Free Edition agota sus recursos durante la ingestión de un parquet de alta volumetria? El pipeline debe capturar el error y reportarlo claramente sin fallar silenciosamente.
- Que sucede cuando los parametros del pipeline definidos en la configuracion de LSDP (`catalogoParametro`, `esquemaParametro`, `tablaParametros`) no estan configurados? El pipeline debe lanzar excepcion con un mensaje indicando que esos parametros son obligatorios.

## Requisitos

### Requisitos Funcionales

- **RF-001**: El pipeline LSDP DEBE leer los parametros de configuracion (`catalogoParametro`, `esquemaParametro`, `tablaParametros`, `rutaCompletaMaestroCliente`, `rutaCompletaTransaccional`, `rutaCompletaSaldoCliente`, `rutaSchemaLocationCmstfl`, `rutaSchemaLocationTrxpfl`, `rutaSchemaLocationBlncfl`) via `spark.conf.get("pipelines.parameters.<nombre>")` al inicializar el modulo.
- **RF-002**: El pipeline LSDP DEBE, al inicializar el modulo, invocar `LsdpConexionParametros` para leer la tabla Parametros de Unity Catalog y obtener el diccionario de configuracion que sera capturado por closure en las funciones decoradas.
- **RF-003**: El pipeline LSDP DEBE, al inicializar el modulo, invocar `LsdpConstructorRutas` seis veces: tres veces para construir las rutas completas de los parquets CMSTFL, TRXPFL y BLNCFL (usando `rutaCompletaMaestroCliente`, `rutaCompletaTransaccional`, `rutaCompletaSaldoCliente`), y tres veces para construir las rutas absolutas de schema location de AutoLoader (usando `rutaSchemaLocationCmstfl`, `rutaSchemaLocationTrxpfl`, `rutaSchemaLocationBlncfl`). Todas las invocaciones usan las rutas relativas recibidas como parametros del pipeline y los datos del diccionario de la tabla Parametros.
- **RF-004**: La utilidad `LsdpConexionParametros` DEBE leer la tabla Parametros usando `spark.read.table(f"{catalogo}.{esquema}.{tabla}")`, retornar un diccionario Python `{Clave: Valor}` con todos los registros e imprimir en pantalla los parametros leidos. No valida que claves especificas esten presentes: es una utilidad generica de lectura. Cada script consumidor es responsable de validar las claves que necesita al usarlas y de lanzar excepcion con el nombre de la clave faltante si no se encuentra en el diccionario. No usa JDBC, Secret Scopes ni Azure Key Vault.
- **RF-005**: La utilidad `LsdpConstructorRutas` DEBE construir rutas dinamicas: si `TipoStorage=Volume`, retorna `/Volumes/<catalogoVolume>/<esquemaVolume>/<nombreVolume>/<ruta_relativa>`; si `TipoStorage=AmazonS3`, retorna `s3://<bucketS3>/<ruta_relativa>` (el parametro `prefijoS3` existe en la tabla Parametros por herencia del Incremento 1 pero NO se utiliza en la construccion de rutas S3, segun Constitution v1.1.0). Debe imprimir la ruta construida en pantalla y DEBE lanzar excepcion (`ValueError`) si el tipo no es reconocido, indicando los valores validos (`Volume` o `AmazonS3`).
- **RF-006**: La utilidad `LsdpReordenarColumnasLiquidCluster` DEBE recibir un DataFrame de Spark y una lista de nombres de columnas del liquid cluster, reordenar el DataFrame colocando las columnas del cluster primero en el orden de la lista y el resto en su orden original, y retornar el DataFrame reordenado. Debe lanzar excepcion si alguna columna del cluster no existe en el DataFrame.

### Entidades Clave

- **Streaming Table cmstfl** (`bronce.regional.cmstfl`): Tabla streaming de bronce que acumula historicamente todos los registros del Maestro de Clientes AS400 ingresados. 72 columnas (70 originales del parquet CMSTFL + `FechaIngestaDatos` (TimestampType, generado con F.current_timestamp()) + `_rescued_data`). Liquid Cluster: `FechaIngestaDatos`, `CUSTID`. Acumulacion: append-only historica.
- **Streaming Table trxpfl** (`bronce.regional.trxpfl`): Tabla streaming de bronce que acumula historicamente todas las transacciones AS400 ingresadas. 62 columnas (60 originales del parquet TRXPFL + `FechaIngestaDatos` (TimestampType, generado con F.current_timestamp()) + `_rescued_data`). Liquid Cluster: `TRXDT`, `CUSTID`, `TRXTYP`. Acumulacion: append-only historica.
- **Streaming Table blncfl** (`bronce.regional.blncfl`): Tabla streaming de bronce que acumula historicamente todos los registros de saldos AS400 ingresados. 102 columnas (100 originales del parquet BLNCFL + `FechaIngestaDatos` (TimestampType, generado con F.current_timestamp()) + `_rescued_data`). Liquid Cluster: `FechaIngestaDatos`, `CUSTID`. Acumulacion: append-only historica.
- **Utilidad LsdpConexionParametros**: Funcion que lee la tabla Parametros de Unity Catalog y retorna un diccionario Python `{Clave: Valor}` con la configuracion del proyecto. Reutilizable en todos los scripts del LSDP.
- **Utilidad LsdpConstructorRutas**: Funcion que recibe el diccionario de parametros y una ruta relativa, y construye la ruta completa de almacenamiento segun `TipoStorage` (Volume o S3). Reutilizable en todos los scripts del LSDP.
- **Utilidad LsdpReordenarColumnasLiquidCluster**: Funcion que recibe un DataFrame de Spark y una lista de nombres de columnas del liquid cluster, y retorna el DataFrame con los campos del cluster como las primeras columnas. Garantiza la calidad del schema de todas las streaming tables de bronce.
- **Tabla Parametros**: Tabla Delta existente en Unity Catalog (creada en el Incremento 1) con columnas `Clave` y `Valor`. Provee la configuracion dinamica del pipeline: tipo de storage, datos de Volume, datos de S3, nombres de catalogos y directorios.
